# Quest Snapshot vs Session Snapshot Explainer v0.1

## Purpose

This short explainer clarifies the difference between:

- V12 Session Snapshot
- Entry Quest Snapshot
- Full Handoff / Handoff Header

It is documentation only. It does not create `outputs/quest_snapshot.md`, implement Quest Snapshot auto-generation, add runtime behavior, call AI or external APIs, score projects, compare snapshots, add hooks/MCP/plugins, or export visuals.

## Quick Difference

| Artifact | Level | Use it when | Current creation path | Not this |
| --- | --- | --- | --- | --- |
| V12 Session Snapshot | Session / day / work-unit | You need to restart recent work without re-explaining routine context. | Local command writes `outputs/session_snapshot.md` when run. | Not completion, not full handoff, not Quest Snapshot auto-generation. |
| Entry Quest Snapshot | Project / quest | You need a project-level As-of re-entry capsule with Gate, next action, unknowns, and boundaries. | Prompt-first / manual workflow using the generator prompt or template. | Not runtime-generated by this repo today. |
| Full Handoff / Handoff Header | Transfer of ownership | A receiving AI or collaborator must know exactly what it now owns before continuing. | Written explicitly by the sender or receiver. | Not satisfied by `outputs/session_snapshot.md` alone. |

## V12 Session Snapshot

A V12 Session Snapshot is a restartable checkpoint for a session, day, or bounded work-unit.

Use it when:

- you are ending a work session
- you want the next AI/Codex to see what changed
- you need to preserve incomplete work without claiming completion
- routine git status, latest commit, current Gate, and HOLD boundaries should be visible

Required distinction:

```text
Session Snapshot is a restartable checkpoint for a session/day/work-unit.
It is not a completion claim.
It is not a full handoff.
It is not Quest Snapshot auto-generation.
```

Current allowed output:

```text
outputs/session_snapshot.md
```

Current command:

```bash
python3 -B tools/save_session_snapshot.py --note "End of session checkpoint"
```

## Entry Quest Snapshot

An Entry Quest Snapshot is a project-level As-of artifact.

Use it when:

- a project, repo, app, workflow, or idea needs a durable re-entry capsule
- the next AI/Codex must understand the Current Gate, one next action, do-not-do boundary, recheck condition, and Completion Line
- unknown evidence should be preserved instead of filled with invented certainty
- the human Decision Owner needs Seat-preserving project context

Current creation path:

- paste `prompts/quest_snapshot_generator_prompt_v0_1.md` into Codex/AI
- or fill `templates/quest_snapshot_template_v0_1.md` manually
- store the result in Codex memory, project context, local notes, or the target repo when appropriate

Current boundary:

```text
Entry Window Radar does not currently auto-generate `outputs/quest_snapshot.md`.
Quest Snapshot runtime generation remains HOLD.
```

## Full Handoff / Handoff Header

A Full Handoff or Handoff Header is the explicit transfer layer.

Use it when:

- another AI/Codex is receiving responsibility for continuing a project
- the receiver must state what it now owns
- the handoff needs to preserve Gate, next action, do-not-do boundary, recheck condition, Completion Line, and Decision Owner

A useful Handoff Header should include:

- what the receiving AI now owns
- Current Gate
- one next action
- do-not-do boundary
- recheck condition
- Completion Line
- who owns the Seat

`outputs/session_snapshot.md` can support a handoff, but it is not enough by itself for project-level ownership transfer.

## Decision Rule

Use Session Snapshot when the question is:

```text
What happened in this work-unit, and how can the next AI restart?
```

Use Quest Snapshot when the question is:

```text
Where is this project now, and what should the next AI understand before continuing?
```

Use a Full Handoff / Handoff Header when the question is:

```text
What does the receiver now own, and what must it preserve?
```
